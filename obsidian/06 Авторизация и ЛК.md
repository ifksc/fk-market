# Авторизация и ЛК

**Статус:** в работе (2026-05-13)
**Зависит от:** Sanctum (уже подключен), `users` таблица (есть), `admin@fk.market` (есть)

> [!note] Документ — это исходный план. Фактическая реализация (2026-05-14/15) местами отклонилась — см. [[Журнал решений#2026-05-15]]:
> - Telegram сделан как **OIDC + PKCE** (id.vk.com style), а не Login Widget — Telegram пометил Widget deprecated.
> - VK ID тоже **OIDC + PKCE** через `id.vk.com/oauth2/auth` — без `socialiteproviders/vkontakte`. Реализовано напрямую руками для консистентности с Telegram'ом.
> - Endpoints у нас: `POST /api/auth/oauth/{provider}/exchange` (а не `/redirect` + `/callback` как было в плане). PKCE-flow обрабатывается на фронте, бэк только меняет code на id_token.
> - JWKS у обоих провайдеров кэшируется в Redis на час (`telegram_jwks`, `vk_jwks`).

## Цели

1. Покупатель может зарегистрироваться/войти, чтобы видеть свои заказы и повторно копировать коды.
2. Прошлые гостевые заказы с тем же email — автоматически подтягиваются в ЛК после первого входа.
3. Админка работает поверх той же таблицы `users` через поле `role` — никакого дублирования.
4. На этом этапе **без** баланса, бонусов, отзывов, Telegram-уведомлений — только email-флоу. Эти фичи приедут отдельной фазой.

## Out of scope (явно отложено)

- Внутренний баланс / пополнение через FK / реферальные программы / кэшбек.
- Уведомления в Telegram-бота и Web Push.
- Отвязка OAuth-аккаунтов и удаление профиля (GDPR-flow).
- Реальная двухфакторная аутентификация.
- Регистрация продавцов (v2).

## 1. Способы входа

| Способ | Для MVP |
| --- | --- |
| Email + пароль | ✅ |
| OAuth: **VK ID** | ✅ |
| OAuth: **Яндекс ID** | ✅ |
| OAuth: **Telegram Login Widget** | ✅ |
| Google OAuth | ❌ (потом) |
| Magic link | ❌ (потом) |
| 2FA | ❌ (потом) |

**Поведение OAuth:**
- Если у пользователя в БД уже есть `users.email` равный email от провайдера — просто привязываем `provider_*` колонки к этому user.
- Если email с этого провайдера получить нельзя (Telegram не отдаёт email) — создаём user без email и просим ввести его в первой же сессии. Без email доступ к покупкам и почте ограничен (см. §5).

## 2. Email-верификация

**Обязательна** для покупателей, зарегистрированных через email/пароль.

Поток:
1. Регистрация → `users.email_verified_at = NULL`.
2. Шлём письмо со ссылкой `/verify-email?token=...` (TTL 24ч).
3. До подтверждения пользователь:
   - видит ЛК с баннером «Подтвердите почту» и кнопкой «Отправить письмо ещё раз»;
   - **может оплачивать** (выдача всё равно идёт на этот email — это и есть проверка по факту), но
   - **не может** сменить email в профиле или зайти в споры до подтверждения.
4. После клика на ссылку — `email_verified_at = now()`, баннер пропадает.

Для OAuth-логинов email считаем подтверждённым, если провайдер сам помечает его как `verified` (VK ID и Яндекс ID — да; Telegram — email вообще нет).

## 3. БД — миграции

### 3.1. `users` — добавить колонки

```sql
ALTER TABLE users
    ADD COLUMN role ENUM('customer','admin','seller') NOT NULL DEFAULT 'customer' AFTER email,
    ADD COLUMN name VARCHAR(120) NULL AFTER role,
    ADD COLUMN phone VARCHAR(32) NULL AFTER name,
    ADD COLUMN email_verified_at TIMESTAMP NULL AFTER phone,
    ADD COLUMN last_login_at TIMESTAMP NULL,
    ADD COLUMN last_login_ip VARCHAR(45) NULL;
```

`admin@fk.market` получает `role='admin'`, остальные — `customer` (default). Текущий middleware `EnsureAdminMiddleware` теперь проверяет `$user->role === 'admin'`.

### 3.2. `user_identities` — связь user ↔ OAuth-провайдеры

```sql
CREATE TABLE user_identities (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT UNSIGNED NOT NULL,
    provider ENUM('vk','yandex','telegram') NOT NULL,
    provider_user_id VARCHAR(64) NOT NULL,    -- ID юзера у провайдера
    provider_email VARCHAR(190) NULL,
    raw_payload JSON NULL,                    -- что отдал провайдер
    created_at TIMESTAMP NULL,
    UNIQUE KEY uniq_provider_uid (provider, provider_user_id),
    INDEX idx_user (user_id),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
```

Это вместо колонок `vk_id/yandex_id/telegram_id` прямо в users — чище и расширяемо.

### 3.3. `email_verifications`

```sql
CREATE TABLE email_verifications (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT UNSIGNED NOT NULL,
    token VARCHAR(64) NOT NULL UNIQUE,
    expires_at TIMESTAMP NOT NULL,
    used_at TIMESTAMP NULL,
    created_at TIMESTAMP NULL,
    INDEX idx_user (user_id)
);
```

### 3.4. `password_resets`

Используем стандартную таблицу Laravel (`php artisan make:password-reset`).

### 3.5. `support_tickets` — споры/жалобы

```sql
CREATE TABLE support_tickets (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT UNSIGNED NULL,                -- может быть NULL если открыто гостем по public_number
    order_id BIGINT UNSIGNED NULL,
    order_item_id BIGINT UNSIGNED NULL,
    kind ENUM('code_not_working','wrong_item','other') NOT NULL,
    subject VARCHAR(200) NOT NULL,
    body TEXT NOT NULL,
    status ENUM('open','in_progress','resolved','rejected') NOT NULL DEFAULT 'open',
    admin_note TEXT NULL,
    created_at TIMESTAMP NULL,
    resolved_at TIMESTAMP NULL,
    INDEX idx_user (user_id),
    INDEX idx_status (status)
);
```

Простой минимум — без переписки и attachments (это уже отдельная фаза).

### 3.6. Привязка гостевых заказов

В `orders` уже есть `user_id NULL`. Дополнительной колонки не нужно. Логика:
- При успешной верификации email (или сразу при OAuth, если email есть и подтверждён провайдером) — выполняем `UPDATE orders SET user_id = :uid WHERE user_id IS NULL AND email = :email`.
- Тот же UPDATE — при подтверждении смены email в профиле.

### 4.3. Админка

`EnsureAdminMiddleware` проверяет `$request->user()?->role === 'admin'`. Все существующие admin-роуты остаются как есть.

Новое:
- `GET /api/admin/users` — список с фильтром (роль / email).
- `PATCH /api/admin/users/{id}` — смена роли, бан.
- `GET /api/admin/tickets` + `PATCH /api/admin/tickets/{id}` — обработка.

## 5. OAuth — детали

Используем `laravel/socialite` + соответствующие community-провайдеры:
- `socialiteproviders/vkontakte` (VK ID)
- `socialiteproviders/yandex` (Яндекс ID)
- Для Telegram — собственный обработчик через Login Widget (проверка HMAC от @BotFather).

Конфиг — отдельные секции в `config/services.php`:
```php
'vk' => ['client_id' => env('VK_CLIENT_ID'), 'client_secret' => env('VK_CLIENT_SECRET'), 'redirect' => env('APP_URL').'/api/auth/oauth/vk/callback'],
'yandex' => [...],
'telegram' => ['bot_token' => env('TELEGRAM_BOT_TOKEN')],
```

Поток для VK/Яндекс:
1. Фронт открывает `/api/auth/oauth/vk/redirect` → 302 на VK.
2. После согласия VK редиректит на `/api/auth/oauth/vk/callback?code=...`.
3. Бэк меняет code на токен, тянет user info → `findOrCreateUserByProvider()` → открывает session → редирект на `/account`.

Поток для Telegram:
1. На фронте кнопка `<script src="https://telegram.org/js/telegram-widget.js?22" data-telegram-login="fkmarket_bot" data-auth-url="/api/auth/oauth/telegram/callback">`.
2. Telegram постит на наш callback подписанный payload (`auth_date`, `hash`).
3. Проверяем HMAC по `bot_token` → создаём/привязываем user.

**Особенность Telegram:** email там нет. Создаём user с `email = NULL`. На первой странице ЛК — обязательный шаг «Введи email».

## 8. Споры/жалобы — UX

На странице заказа (`/account/orders/[public_number]`) под каждым item — кнопка «Код не работает / неверный код». По клику — модалка:
- предзаполнен `order_id` / `order_item_id`,
- выбор `kind` (radio): «Код не работает» / «Пришёл не тот товар» / «Другое»,
- описание (textarea),
- по сабмиту → POST `/api/me/tickets` → редирект на `/account/tickets`.

Заявка создаётся со статусом `open`. В админке у `/admin/queue` (или новой `/admin/tickets`) видны все открытые. Админ может ответить — что меняется на `in_progress` или `resolved`, и письмо уходит юзеру.

SLA в `01 ТЗ.md` уже есть: 14 дней на замену.

## 9. Привязка прошлых гостевых заказов

При любом событии «у user'а появился подтверждённый email» (регистрация + верификация / OAuth с verified email / смена email):

```php
Order::whereNull('user_id')
    ->where('email', $user->email)
    ->update(['user_id' => $user->id]);
```

Это решение покрывает все случаи. Никакого ручного «введите номер заказа + email» — оно есть в Roadmap, но не в текущей фазе.

## 10. Безопасность

- Пароли — `bcrypt` (Laravel default, 12 rounds).
- Login rate-limit: `5 попыток / 1 мин на IP+email` (через `RateLimiter`).
- Verify-email и reset-password токены — `Str::random(64)`, single-use, TTL 24ч.
- Сессии — Sanctum SPA с `SameSite=Lax`, `HttpOnly`, `Secure`.
- CSRF — стандартный механизм Laravel (cookie `XSRF-TOKEN` + заголовок `X-XSRF-TOKEN`).
- В админских роутах — проверка `role='admin'` через middleware (как сейчас).
