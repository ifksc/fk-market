# 08 Динамические цены — ТЗ

Доработка раздела «Наценки»: цена товара меняется от динамики спроса.
Статус: **план на ревью, код не начат.**

## 1. Идея

Поверх статической цены — надбавка от спроса. Товар, который активно
покупают, временно дорожает; когда спрос спадает — цена пошагово возвращается
к базовой. Настраивается по категориям.

## 2. Принятые решения (обсуждение 2026-05-18)

- Спрос измеряется **по каждому товару** (его собственные продажи). Настройки
  (порог/шаги/потолок) — общие на категорию.
- Порог — **относительный**: продажи за окно сравниваются со скользящим
  средним этого же товара.
- Снижение — **шаговое**: каждый «тихий» период `dynamic_pct` − шаг.
- Пересчёт — **каждые 4 часа** (крон).
- Динамика только **вверх**: `dynamic_pct ∈ [0, max]`, ниже базовой не опускаем.

## 3. Модель цены

Сейчас: `price_final = price_base × (1 + наценка/100)`.

Становится:
```
price_final = price_base × (1 + наценка/100) × (1 + dynamic_pct/100)
```
`dynamic_pct` — новое поле товара (0 по умолчанию), меняет крон-джоб.

## 4. Модель данных

**`products`** — новая колонка:
- `dynamic_pct` DECIMAL(5,2) DEFAULT 0 — текущая надбавка спроса.

**`dynamic_pricing_settings`** — настройка на категорию:

| Поле | Тип | Назначение | Дефолт |
|------|-----|-----------|--------|
| `category_id` | FK, unique | категория | |
| `is_active` | bool | включена ли динамика | false |
| `window_hours` | int | окно «недавнего спроса» | 24 |
| `baseline_days` | int | период скользящего среднего | 14 |
| `trigger_pct` | decimal | на сколько % выше базлайна продажи запускают рост | 50 |
| `min_sales` | int | минимум продаж в окне, иначе товар не трогаем | 3 |
| `raise_step_pct` | decimal | шаг повышения | 5 |
| `lower_step_pct` | decimal | шаг снижения | 5 |
| `max_pct` | decimal | потолок надбавки | 20 |

Числа — дефолты; админ настраивает на категорию. Настройка применяется к
прямой `category_id` товара (без наследования от родителя — в v1 проще;
наследование — возможное развитие).

## 5. Алгоритм пересчёта (крон, каждые 4 часа)

Команда `pricing:apply-dynamic`. Для товаров в категориях с `is_active`:

1. `recent` — продано единиц товара за `window_hours` (из `order_items` +
   `orders.paid_at`, только оплаченные/не отменённые/не возвраты).
2. `baseline` — ожидаемые продажи за окно из скользящего среднего:
   `(продажи за baseline_days) ÷ baseline_days × (window_hours / 24)`.
3. **Рост:** если `recent ≥ min_sales` И `recent ≥ baseline × (1 + trigger_pct/100)`
   → `dynamic_pct = min(dynamic_pct + raise_step_pct, max_pct)`.
4. **Снижение:** иначе → `dynamic_pct = max(dynamic_pct − lower_step_pct, 0)`.
5. Пересчитать `price_final`.

«Храповик»: при устойчивом спросе цена ползёт вверх по шагу за период, при
затишье — так же пошагово вниз до базы.

`min_sales` — защита от ложных срабатываний: у нишевого «мёртвого» товара одна
продажа формально «бесконечно выше базлайна», но без минимума абсолютной
активности рост не запускаем.

Производительность: продажи берём двумя агрегирующими запросами
(`GROUP BY product_id`) — на окно и на baseline, не по запросу на товар (как
в `PriceRecomputer` для каталога 10k+). Работа батчами.

## 6. Backend

- Миграции: `dynamic_pct` в `products`; таблица `dynamic_pricing_settings`.
- Модель `DynamicPricingSetting` (`belongsTo` Category).
- `services/DynamicPricingService` — расчёт `dynamic_pct`.
- `console/ApplyDynamicPricingCommand` (`pricing:apply-dynamic`) — крон-джоб.
- Планировщик в `AppServiceProvider::boot` — `everyFourHours()` +
  `withoutOverlapping()`.
- **`PriceCalculator` / `PriceRecomputer` дорабатываются:** `price_final`
  включает множитель `(1 + dynamic_pct/100)`. Иначе сохранение товара и синк
  провайдера (`ProductRefresher`) затрут динамику.
- `Admin/DynamicPricingController` — CRUD настроек по категориям.
- Роуты — в admin-группе (`admin`).

## 7. Админка

Раздел «Наценки» (`/admin/pricing`) — новая вкладка «Динамические цены»:
список категорий, по каждой — настройки (вкл/выкл, окно, порог, шаги, потолок).
В списке/карточке товаров админки — показывать текущий `dynamic_pct`, чтобы
видеть, какие товары сейчас «разогреты».

## 8. Витрина

`price_final` уже включает динамику — отдельных изменений витрины не требуется.
Опционально: бейдж «🔥 Популярно» при `dynamic_pct > 0` (решить на ревью).

## 9. Известное ограничение — цена при оплате

Checkout считает цену из БД на момент оформления (защита от подделки с фронта).
Если динамика пересчитается между просмотром и оплатой — покупатель увидит
одну цену, заплатит другую. При пересчёте раз в 4 часа и коротком ISR-кэше
окно расхождения мало — для v1 принимаем как есть. На будущее — фиксировать
показанную цену на короткий TTL либо явно показывать актуальную цену на чекауте.

## 10. Фазы

- **Ф1 (бэкенд):** миграции + модель + `DynamicPricingService` + команда +
  крон + доработка `PriceCalculator`/`PriceRecomputer`. Категории по умолчанию
  выключены — динамика не влияет, пока админ не включит.
- **Ф2 (админка):** вкладка «Динамические цены» + показ `dynamic_pct` в товарах.
