# Журнал решений

Дневник принятых решений и их обоснований. Формат: `ЧЧЧЧ-ММ-ДД` + короткая запись. Помогает через полгода понять «почему мы сделали именно так».

## 2026-04-23

### 🎯 Выбран стек: Laravel 11 + MySQL 8 + Next.js 14 (+ Redis)
**Почему:** PHP/MySQL — привычный стек владельца проекта («у нас везде MySQL»). Next.js даёт SSR для SEO каталога. Redis — для очередей (выдача) и кэша.

### 🎯 Ниша магазина: нейтральная, без 18+
**Почему:** проще с рекламой, платёжными провайдерами, приёмом карт. Товары — VPN, скины Steam, ИИ-аккаунты, ключи, подписки, услуги.

### 🎯 Заказчик — физлицо, чеки не нужны
**Почему:** старт с минимальной юридической нагрузкой. При росте оборота — оформляем самозанятость/ИП и подключаем кассу (54-ФЗ).

### 🎯 Telegram-бот — вне MVP
**Почему:** не нужен для MVP, добавится в v2 (нотификации админа, опционально — выдача кодов в TG).

### 🎯 OAuth: VK ID, Яндекс ID, Telegram Login Widget
**Почему:** основная аудитория в РФ, эти три провайдера перекрывают большинство.

### 🎯 Добавлена категория «ИИ-аккаунты»
**Что включает:** ChatGPT Plus/Team, Claude Pro, Midjourney, Gemini Advanced, Perplexity Pro, API-ключи OpenAI/Anthropic.
**Риск зафиксирован:** по ToS OpenAI/Anthropic продажа аккаунтов не разрешена — в оферте прописать гарантию работоспособности N дней и заложить бюджет на замены.

### 🎯 Схема БД: 18 бизнес-таблиц, `seller_id` везде с v1
**Почему:** мультивендорность v2 не будет ломать БД. В v1 всё привязано к системному продавцу `platform`.

### 🎯 Режимы выдачи: stock / api / manual + fallback
**Почему:** покрывает все типы товаров одной моделью. Универсально расширяется.

### 🎯 Шифрование: Laravel `Crypt` (AES-256, ключ в `APP_KEY`)
**Почему:** стандартное решение фреймворка, ключ не в БД (защита при утечке дампа).

### 🎯 Поиск: MySQL FULLTEXT на v1, Meilisearch при 50k+ товаров
**Почему:** не раздуваем стек. Переход на Meilisearch — 2 дня работы, когда реально понадобится.

### 🎯 Тикеты поддержки — НЕ в MVP
**Почему:** хватит email + формы «написать в поддержку». Полноценную систему подключим по потоку обращений.

### 🎯 Логотип — концепция A (Squircle)
**Почему:** универсально работает в разных размерах (40px шапка, 24px favicon), градиент в бренде.

### 🎯 Стиль дизайна — современный гибрид с переключателем темы
**Почему:** часть аудитории любит dark (геймерская аудитория), часть — light. Тогглим.

### 🎯 Продакшн-сервер: `194.87.118.214` (SSH `22`)
**Почему:** есть в распоряжении, не тратим время на выбор VPS-провайдера и онбординг.
**История:** изначально планировался `5.42.99.62`, но сменён на `194.87.118.214` после возврата к проекту.
**Последствия:** на этапе [[Roadmap|Фазы 4]] идём настраивать этот сервер — Docker, nginx, TLS, бэкапы. DNS `fk.market` направляем A-записью туда же. CI/CD при желании подключаем к тому же IP.
**Примечание:** это НЕ сервер для Obsidian LiveSync — у LiveSync свой CouchDB на другом IP.

### 🎯 Админка готова в полном объёме
**Дата:** 2026-05-08
**Что сделано:**
- Sanctum-авторизация админа (POST /api/admin/login → токен в localStorage)
- EnsureAdminMiddleware: только role=admin может попасть в /api/admin/*
- Бэк-эндпоинты на все основные сущности:
  - Товары: GET / POST / PUT / DELETE (архив), включая загрузку 4 типов фильтров и сортировок
  - Склад ключей: GET / POST (массовая загрузка) / DELETE; payload шифруется на лету; в API возвращается только preview
  - Заказы: GET / GET по id; actions: cancel, refund, redeliver (повторная отправка email), refulfill (перезапуск выдачи)
  - Очередь выдачи: GET с фильтрами (open/mine/overdue/done/failed); actions: claim, complete (с payload+комментарием), cancel
  - Поставщики API: GET список / GET one / PUT настройки (включая credentials через Crypt-cast)
  - Pricing rules: GET / POST / PUT / DELETE — кроме global, его только апдейт
  - Settings (key-value): GET / PUT (массово несколько ключей)
- Фронт: каждая страница админки рабочая (`/admin/*`), общий layout с sidebar и проверкой токена; общий Header сайта на /admin не показывается (через клиентский SiteShell)
- Service `PriceCalculator`: пересчитывает price_final при сохранении товара исходя из markup_pct → или применимого pricing rule
- В bootstrap/app.php зарегистрирован alias 'admin' для middleware

**Грабли пути:**
- `Application::configure()->withMiddleware()` в Laravel 11 — alias регистрируется через `$middleware->alias([...])`. Через regex проще не патчить, проще `str.replace`.
- Heredoc `<<'PY'` через ssh для python-патча работает, но `re.sub` ругается на backslash escapes в replacement. Безопаснее `str.replace` или lambda.
- Если контроллер запрашивает в `__invoke` сервис со скалярными параметрами в конструкторе — Laravel-контейнер не сможет автомatically разрешить. Регистрация singleton в AppServiceProvider обязательна.
- Header сайта (root layout) показывался поверх админки. Решено клиентским компонентом `SiteShell`, который проверяет `pathname.startsWith('/admin')` и не рендерит Header/Footer.

### 🎯 MVP работает целиком — полный путь покупки в проде
**Дата:** 2026-05-08 ночь
**Что сделано:**
- POST `/api/checkout` принимает корзину, создаёт `orders` + `order_items` транзакционно, формирует платёжную ссылку Free-Kassa с MD5-подписью на `secret_word_1`
- POST `/api/payments/fkwallet/webhook` проверяет MD5 на `secret_word_2`, меняет статус заказа на `paid`, запускает `FulfillOrderJob`
- `FulfillOrderJob` (sync queue): для `fulfillment_mode=stock` — резерв первого `stock_items`, копия `payload` в `order_items.delivered_payload`, status=`delivered`. Для `api/manual` — создание `fulfillment_tasks` (api пока заглушка под этап 2)
- Email с кодами → `MAIL_MAILER=log` (письма пишутся в `storage/logs/laravel.log`)
- Frontend: `/checkout` делает реальный POST, редирект на pay.freekassa.com; `/checkout/success` — polling статуса, копирование выданного кода; `/checkout/failed` — отмена
- В ЛК Free-Kassa прописаны Result/Success/Failure URLs + IP whitelist `194.87.118.214`

**Проверено:** реальная покупка через `https://fk.market` отрабатывает end-to-end.

**Грабли пути:**
- `useSearchParams()` в Next.js 16 требует `<Suspense>` обёртку для prerender (на странице `/checkout/success`).
- Service автоматически не разрешается через DI, если конструктор требует примитивы (string). Решение: `singleton` в `AppServiceProvider`.
- Heredoc через ssh + zsh ломается при странном символе в начале — лучше bash-скрипт через scp вместо длинных inline-команд.

**Что ещё впереди:**
- Этап 2: интеграция с внешним API товаров (для `fulfillment_mode=api`) — синхронизация каталога и автовыдача через провайдера. Нужна документация конкретного API.
- Реальная авторизация (Sanctum + OAuth: VK, Яндекс, Telegram).
- Реальные данные в `/account` (вместо демо).
- Admin UI — управление товарами, заказами, очередью выдачи.
- SMTP для писем вместо `log`-драйвера.
- Картинки товаров вместо градиентов-заглушек.
- Git-репозиторий + GitHub Actions для автодеплоя.
- Бэкапы БД.
- Замена `php artisan serve` на FPM+nginx-Docker (сбросится docker hub лимит).

### 🎯 Фронт деплоен — полный стек в проде на одном сервере
**Дата:** 2026-05-08 ночь
**Что сделано:**
- Next.js 16 + React 19 + Tailwind 4: главная, каталог, карточка товара, корзина (localStorage), оформление, login (OAuth-заглушки), личный кабинет
- Standalone Docker-образ (~150 МБ), запущен через `docker compose` в `/opt/fk.market/frontend/`

- nginx проксирует:
  - `/api/*` → Laravel (127.0.0.1:8000)
  - `/_next/static/*` → Next.js с кэшем 365d (immutable)
  - `/*` → Next.js (127.0.0.1:3000)
- Live: `https://fk.market/` отдаёт Next.js, `https://fk.market/api/*` — Laravel JSON
- TLS остался прежний — Cloudflare Origin Certificate + Full (strict)

**Архитектура runtime:**
```
Cloudflare proxy → Nginx (TLS, /etc/ssl/fkmarket/origin.{pem,key})
  ├── /api/  → Laravel в Docker (php artisan serve, временно)
  └── /     → Next.js в Docker (standalone, node server.js)
```

**Что ещё впереди:**
- Реальная авторизация (sign-up / login через POST /api/auth/...)
- Реальное создание заказа через POST /api/checkout
- Интеграция FKwallet (платёжная ссылка + webhook)
- Замена `php artisan serve` на FPM+nginx-Docker (когда сбросится Docker Hub лимит)

### 🎯 Сервер настроен и API запущен на проде
**Дата:** 2026-05-07 ночь (в течение одного вечера от голого VPS до live-API)
**Что сделано:**
- Bootstrap сервера `194.87.118.214` (Ubuntu 24.04 в Yandex Cloud, hostname `msk-1-vm-yc37`): Docker, ufw 22/80/443, fail2ban, nginx, certbot, пользователь `deploy` без пароля
- Laravel 11 в Docker (php 8.3 + mysql 8 + redis 7), пока через `php artisan serve` (вместо FPM-контейнера nginx — он не скачался из-за Docker Hub rate-limit, отложили на потом)
- Миграции 18 таблиц, сиды: 7 категорий + 11 товаров (ChatGPT Plus, NordVPN, AK-47 Redline и т.п.)
- Системный nginx как reverse proxy на 127.0.0.1:8000
- TLS: Cloudflare Origin Certificate (15 лет), Full (strict)
- DNS: Cloudflare proxy (оранжевое облачко) для `fk.market`

**Live URLs:**
- https://fk.market/api/health
- https://fk.market/api/categories
- https://fk.market/api/products

**Грабли пути и чему научились:**
- `setup.sh` писал `migrate:fresh --seed --force` — на проде такое НЕЛЬЗЯ, удаляет данные. Сейчас пустая БД, не критично, но в `deploy.sh` уже `migrate --force` без `:fresh`.
- В моделях был несуществующий `AsEncryptedString::class` — заменено на `'encrypted'` (строковый cast). Поправлено и локально, и на сервере через `sed`.
- nginx 1.24 не поддерживает `http2 on;` отдельной директивой — пишется `listen 443 ssl http2;` в одной строке.
- Docker Hub лимит на анонимные pull-ы (~100 за 6 часов) — забивается быстро при двух полных перезапусках setup.sh. Решение для прода: `docker login` под бесплатным аккаунтом (увеличивает до 200/6ч) либо использовать зеркало.
- Долгие команды через ssh нужно запускать в `tmux` — иначе SSH disconnect рвёт процесс.

## 2026-05-19

### 🐛 Архивный товар можно было купить (PR #136)
Карточка `archived`/`draft` товара открывается по прямой ссылке (`/products/{slug}` route-binding не фильтрует статус) и позволяла класть товар в корзину. `/checkout` на бэке такие товары и так отклонял (`status=active`), но фронт не знал. Решение: `transform` отдаёт `status`; в `ProductBuyBox` при `status != active` кнопки покупки заменяются плашкой «Товар снят с продажи». Карточка остаётся видимой (по требованию).

### 🎯 Авто-скидка при снижении цены поставщика (PR #137)
При синке цен FKwallet: цена товара снизилась → прежняя пишется в `price_old`, на витрине показывается скидка; выросла до уровня `price_old` → скидка убирается. Логика `ProductRefresher::adjustPriceOld`: сравниваем витринную `price_final`; `price_old` хранит максимум недавней цены (серия снижений не занижает скидку); чистим только когда цена ≥ `price_old`; порог 1% против дребезга. Для синк-товаров `price_old` целиком под авто-логикой — поле в админке заблокировано. Договорённость: бейдж «🔥 Хит» (будущие динамические цены) приоритетнее бейджа скидки — зафиксировано в `08 Динамические цены.md`.

Чинит кейс «товар неправильно выгрузился».

### 🐛 Новый номинал у поставщика → отдельная карточка вместо варианта группы (PR #132)
**Диагноз:** товары группируются «1 leaf-категория FK = 1 Product». `ProductGrouper::groupAll` создавал Product'ы только из непривязанных `provider_products` и не умел дозаливать варианты в уже существующую группу. Номинал, появившийся у поставщика после создания группы, видится как один непривязанный pp → ветка «один товар» → standalone-карточка (кейс `xbox-wallet-code-ssha10` вместо варианта группы `xbox-wallet-code-ssha`).
**Решение (вариант B, консервативный):** `groupAll` дозаливает непривязанные pps в существующую группу (`attachToGroup` + переиспользуемый `syncGroupVariants`); авто-синк ничего не архивирует. Команда `products:merge-into-group {product} {--into} {--dry} {--force}` — разовое слияние уже-ставшего-отдельным товара: pps переносятся в группу, варианты пересобираются, дубль архивируется (`status=archived`, без удаления). Edge вне scope: категория с *одиночным* товаром (не группой) + новый номинал — single→group конверсия не делается, чинится той же командой вручную.
**Применено (2026-05-19):** на проде выполнен `products:merge-into-group xbox-wallet-code-ssha10` — товар $10 слит вариантом в группу `xbox-wallet-code-ssha`, отдельная карточка #1609 заархивирована. ✅

### 🎯 Два бага OAuth/инфра закрыты — state mismatch и fk_frontend unhealthy
**Контекст:** разбор двух открытых багов из [[Bugs]].

**Баг 1 — «Несовпадение state» при OAuth-логине.**
- *Корень:* фронт хранил OAuth-session (`state` + `code_verifier` + `redirect_uri`) в `sessionStorage` под одним фиксированным ключом (`fk-vk-oauth`). Сценарий поломки: клик «Войти через VK» (state A) → страница VK → кнопка «назад» → клик ещё раз (state B перезаписывает A) → через историю браузера завершается **первая** авторизация → callback приходит со state A, а в хранилище уже B → `session.state !== state` → ошибка. Латентно тем же страдали Telegram и Яндекс.
- *Что сделано:* ключ хранения теперь включает `state` — `fk-vk-oauth:<state>`. Незавершённые попытки уживаются в `sessionStorage`, callback находит **свою** по state. Поиск по state сам стал CSRF-проверкой: подделанный/чужой state → ключа нет → null → «сессия истекла» (безопасный исход). Явная проверка `session.state !== state` убрана как ставшая избыточной.
- Применено ко всем 3 провайдерам: `lib/{vk,telegram,yandex}-oauth.ts` + 3 callback-страницы.

**Баг 2 — `fk_frontend` вечно `unhealthy`.**
- *Корень:* healthcheck в `frontend/docker-compose.yml` — `wget ... http://localhost:3000/`. Next.js standalone (`server.js`) слушает только IPv4 (`HOSTNAME=0.0.0.0`), а BusyBox wget в alpine-образе резолвит `localhost` в IPv6 `::1` первым → «Connection refused» → exit 1. Контейнер живой (`curl 127.0.0.1:3000` → 200, `node fetch` → 200), но healthcheck падал.
- *Что сделано:* `localhost` → `127.0.0.1` в healthcheck — форсим IPv4. Проверено на проде: `wget --spider http://127.0.0.1:3000/` → exit 0.

**Грабли пути:**
- Первая гипотеза по багу 2 была неверной — думал, BusyBox wget не понимает `--tries`/`--spider`. Прод-проверка показала, что опции принимаются, а валится именно connect. Урок: не чинить по предположению, сперва воспроизвести на проде.
- Healthcheck — это конфиг контейнера, задаётся при его создании. После деплоя нужен `docker compose up -d` (пересоздание контейнера), просто rebuild образа + `docker restart` новый healthcheck не подхватят.

**Что ещё впереди:**
- Проверить после деплоя, что `fk_frontend` стал `healthy`; если нет — `docker compose up -d` фронту вручную.

### 🎯 Яндекс ID OAuth — реализован тем же паттерном, что VK
**Статус:** развёрнут на проде 2026-05-16. Smoke-тест пройден, финальная проверка реальным входом — за пользователем.
**Контекст:** VK ID запущен на проде — следующий провайдер по плану `/login` и [[Kanban]] — Яндекс ID.

### 🎯 JWKS stale-fallback + усиление retry
**Контекст:** Telegram-логин упал с «Подпись id_token не сошлась». В логе — `cURL error 28 ... jwks.json`, кэш `telegram_jwks` пустой.

**Корень:** JWKS-кэш TTL 1 час. Когда он истекает И именно в этот момент маршрут до `oauth.telegram.org` флакает — refresh не проходит, `fetchOauthJwks` кидал `RuntimeException` → verify падал. При том что JWKS-ключи меняются раз в месяцы — прошлый набор был бы полностью валиден.

**Что сделано:**
- `fetchOauthJwks` — двухуровневый кэш: `$key` (свежий, 1 час) + `$key_stale` (последняя успешная копия, 30 дней). Если сеть не дала валидный JWKS — отдаём stale вместо исключения. Падаем только если нет даже stale.
- retry на всех сетевых вызовах к Telegram/VK усилен 2→3 (4 попытки суммарно): token exchange, user_info, JWKS. При ~40% потерь handshake'ов шанс успеха ≈ 1 − 0.4⁴ ≈ 97% против ~94% на 3 попытках. Это дешёвая альтернатива egress-прокси (прокси держим как план Б, если маршрут начнёт деградировать системно).
- После деплоя прогрел оба кэша вручную (первая проба сети упала, вторая прошла — лишнее подтверждение нестабильности).

**Почему не прокси:** проблема — конкретный сетевой сегмент Yandex Cloud ↔ Telegram CDN. Прокси помог бы только если у прокси-хоста маршрут стабильнее (не гарантия), плюс это +1 компонент и точка отказа. Сначала измеряем частоту и выжимаем retry; прокси — если retry не вытянет.

### 🎯 Route [login] not defined — добили через redirectUsing
**Контекст:** фикс от 2026-05-15 (`renderable(AuthenticationException)`) закрыл не все случаи — `Route [login] not defined` всё ещё падал на части запросов.

**Корень (из stack trace):** `Authenticate::redirectTo()` зовёт `route('login')` **внутри** `unauthenticated()`, ДО того как `AuthenticationException` долетает до renderable-handler'а. Дефолтный `redirectTo()` делает `$request->expectsJson() ? null : route('login')`. Значит:
- запрос **с** `Accept: application/json` → expectsJson=true → null → 401 ✓
- запрос **без** этого заголовка (браузер открыл `/api/*` напрямую, fetch без заголовка) → `route('login')` → `RouteNotFoundException` → 500.

**Что сделано:** `Authenticate::redirectUsing(fn () => null)` в `AppServiceProvider::boot`. Теперь `redirectTo()` никогда не дёргает `route('login')` — сразу чистый `AuthenticationException` → renderable → JSON 401. Проверено: `/api/me`, `/api/me/orders` без auth и без Accept-заголовка → 401.

**Грабли:** `redirectUsing` — статический метод `Illuminate\Auth\Middleware\Authenticate`, git-tracked через AppServiceProvider. Не пришлось трогать не-git-tracked `bootstrap/app.php`.

### 🎯 VK ID — профиль через user_info, JWKS-путь не сработал
**Дата:** 2026-05-16
**Контекст:** применили VK-креды, юзер прошёл flow — упёрлись в «Подпись id_token не сошлась». В логе: `JWKS HTTP 404`.

**Что нашли:** у VK ID **нет публичного JWKS endpoint'а**. Перепробовано (всё 404):
- `https://id.vk.com/oauth2/public_keys` (и `?client_id=`, и POST с body)
- `https://id.vk.com/oauth2/public_key` (singular)
- `https://id.vk.com/.well-known/jwks.json`, `/.well-known/openid-configuration`, `/.well-known/oauth-authorization-server`
- `https://login.vk.com/...`, `https://api.vk.com/method/auth.getPublicKey`

OIDC discovery документа у VK ID нет вообще. Документация на `id.vk.com` недоступна для веб-фетча.

**Решение:** отказались от проверки id_token по JWKS. Вместо этого:
- из ответа token endpoint берём `access_token`;
- зовём `POST https://id.vk.com/oauth2/user_info` с `access_token` + `client_id`;
- получаем профиль `{ user: { user_id, first_name, last_name, email, avatar } }`.

**Почему это безопасно:** `access_token` добыт backend-to-backend по TLS с нашим `client_secret`. Канал доверенный — JWT-верификация id_token нужна в первую очередь когда токен путешествует через браузер. В server-side flow проверка подписи не обязательна. Это стандартный OAuth2 (без OIDC-слоя).

**Что сделано в коде:** `oauthVkExchange` — `id_token`+JWKS+iss/aud-проверки заменены на `access_token`+`user_info`. Убран мёртвый `fetchVkJwks`. `fetchOauthJwks`/`fetchTelegramJwks` остались для Telegram (у него JWKS работает).

**Грабли пути:**
- `POST /oauth2/public_keys` без body отдаёт `411 Length Required` (nginx) — выглядит будто endpoint существует, но с body всё равно 404. Ложный след.
- VK ID `user_info` endpoint требует именно POST: GET даёт `413 Request Entity Too Large` (странная реакция nginx 0.3.33 у VK).

**Итог:** VK ID OAuth работает на проде. Юзер успешно вошёл 2026-05-16.

## 2026-05-15

### 🎯 Telegram OIDC — fast-fail на нестабильный маршрут до oauth.telegram.org
**Дата:** 2026-05-15 ночь
**Контекст:** после миграции с Login Widget на OIDC с PKCE юзер видел `502 API /auth/oauth/telegram/exchange`. Дебаг показал последовательность сюрпризов.

**Что нашли:**
- Маршрут Yandex Cloud (`194.87.118.214`) → Telegram CDN периодически роняет TCP-handshake. В измерении: 2 из 5 подключений к IPv4-эджу `149.154.167.99` уходят в timeout без `CURLINFO_PRIMARY_IP`.
- DNS `oauth.telegram.org` отдаёт и IPv6 (`2001:67c:4e8:f004::9`), и IPv4. CLI `curl` выживает через happy-eyeballs, PHP-биндинг libcurl — нет: при первой неудачной семье висит до full timeout.
- Принудительный `CURL_IPRESOLVE_V4` помогает не везде: эджи IPv4 тоже периодически глушат handshake.

**Что сделано (`AuthController::oauthTelegramExchange`):**
- `connectTimeout(5)` — fast-fail на TCP-handshake'е;
- `timeout(15)` — total cap;
- `retry(2, 500ms)` только на `ConnectionException` (4xx ретраить нельзя, code одноразовый);
- 502 → 503 + `Retry-After: 5` (это transient upstream, не «сервер сломан»).

**Worst case:** ~11 секунд вместо 30+, успех при 60%-ной доле проходящих handshake'ев ≈ 93.6%.

### 🎯 JWKS Telegram → кэш в Redis на час
**Контекст:** после фикса token exchange наткнулись на ту же сетевую нестабильность на JWKS endpoint. Каждый успешный login = 2 сетевых вызова к Telegram → удвоенный шанс залипнуть.

**Что сделано:** `AuthController::fetchTelegramJwks(bool $forceRefresh)` — приватный метод с `Cache::remember`-семантикой:
- 99% запросов берут JWKS из Redis (TTL 1 час);
- если `JWT::decode` упал с `UnexpectedValueException` (kid не нашёлся — могла быть ротация ключей) — `forceRefresh=true`, тянем заново;
- если сеть упала, а кэша нет — `RuntimeException` → 422 «подпись не сошлась».

JWKS у Telegram меняется раз в месяцы — кэширование безопасно.

### 🎯 CI deploy — `docker compose exec -T` съедал stdin heredoc'а
**Контекст:** workflow `Apply backend` показывал `success`, но `docker restart fk_app` не вызывался — `Up N minutes` оставалось от предыдущего ручного рестарта. Из-за этого с 2026-05-07 любой backend-фикс деплоился (файлы в `src/` обновлялись), но **не применялся** — `php artisan serve` live-process кэшировал классы в памяти. Это объясняет некоторые «магические» провалы предыдущих сессий.

**Корень:** в workflow:
```yaml
ssh ... 'bash -s' <<'REMOTE'
  ...
  docker compose exec -T app php artisan migrate --force
  docker compose exec -T app php artisan config:clear
  ...
REMOTE
```
Heredoc прокидывается через stdin ssh. `docker compose exec -T` (даже без TTY) наследует stdin и потребляет его. Первый exec съедал остаток heredoc'а → bash видел EOF → выход с кодом 0 → ssh exit 0 → workflow success без выполнения остатка скрипта.

**Лечение:** `< /dev/null` для каждой `docker compose exec`. После фикса в логах вижу 2 события restart на каждый push в main (`fk_app` + `fk_worker`).

**Долгосрочно:** замена `php artisan serve` на FPM-Docker уберёт зависимость от рестарта — FPM перезапускает воркеры на новые запросы.

### 🎯 Авторизация и ЛК — pending_email теперь в /api/me
**Контекст:** OAuth-юзер (Telegram, `email=null`) вводил email в `/account/profile`, получал плашку «отправили код на X», но ссылка «подтверждения почты» на `/verify-email` редиректила его обратно — охранник `user.email === null → /profile` срабатывал, не зная о pending'е.

**Что сделано:**
- `AuthController::serializeUser` отдаёт `pending_email` — последний неиспользованный, не истёкший `EmailVerification.new_email`;
- `/verify-email`: охранник пропускает если `pending_email` есть; в шапке показывает `pending_email ?? user.email`;
- `/account/profile EmailSection`: плашка про код берётся из `user.pending_email` (переживает refresh страницы);
- `EmailVerification::issue` — перед созданием новой записи инвалидирует все unused юзера (`used_at=now`). Юзер может указать другой email до подтверждения первого; старый код перестаёт работать;
- На `/verify-email` добавил «Указать другой» → ведёт на `/account/profile?need=email`.

### 🎯 /api/* возвращает 401 JSON на unauth
**Контекст:** `curl /api/me` без Bearer выдавал `500 + Route [login] not defined` — Laravel дефолт редиректит на web-роут `login`, которого у нас нет.

**Что сделано:** `AppServiceProvider::boot` регистрирует `renderable(AuthenticationException)` — для `$request->is('api/*')` или `expectsJson()` возвращаем `{"message":"Unauthenticated"}` со статусом 401. Web-роуты при появлении сохранят дефолтное поведение.

### 🎯 Resend verification использует pending_email + защита от null
**Корень:** `resendVerification` вызывал `EmailVerification::issue($user)` без второго аргумента → запись с `new_email=null` → `VerifyEmailMail::envelope()` падал с `read property "address" on null` у OAuth-юзеров без email. Плюс регресс от предыдущей инвалидации: resend обнулял pending у OAuth-юзеров.

**Что сделано:** перед `issue()` достаём текущий pending'овый `new_email` и передаём его. Если ни pending, ни `user->email` — 422 «Сначала укажите email в профиле» вместо падающей почты.

### 🎯 Cloudflare перехватывает 5xx — auth-ошибки теперь 4xx
**Дата:** 2026-05-15 после применения VK кредов
**Контекст:** запустил VK-логин на проде. Smoke test возвращал `error code: 502` от Cloudflare вместо нашего JSON. В laravel.log при этом видны WARNING'и `VK OAuth missing id_token`, `device_id is invalid` — то есть код работал, доходил до VK, VK честно отвечал, а наш контроллер на эту ситуацию возвращал HTTP 502.

**Корень:** Cloudflare по умолчанию **перехватывает 5xx от origin** и подменяет тело на свою красно-серую error-page. Любой `response()->json([...], 502/503/504)` из Laravel доходит до фронта в виде HTML «error code 502». Юзер не видит наш `message`.

**Что сделано:** в `oauthVkExchange` и `oauthTelegramExchange`:
- Явная проверка `$resp->json('error')` в дополнение к HTTP-статусу (VK любит отдавать 200 + JSON с error-полем — это надо считать failure, не успехом);
- «missing id_token» переведено с 502 на 422 — Cloudflare его не трогает.

**Что осталось на 5xx (правильно):** случаи когда мы реально не дозвонились до провайдера (`ConnectionException`) — там 503 + `Retry-After`. Это transient и Cloudflare-перехват тут даже полезен (юзер видит что что-то с сетью).

**Грабли пути:** оставшийся 502 в коде для условия «id_token пустой при 200 OK» был артефактом — этот код почти невозможно достичь у Telegram, поэтому проблема всплыла только сейчас на VK.

### 🎯 VK ID OAuth — реализован тем же паттерном, что Telegram OIDC
**Дата:** 2026-05-15 утро
**Контекст:** Telegram-логин стабилизировали — пришло время добавить VK (план был отмечен как «следующий этап» в `/login` и в [[06 Авторизация и ЛК]]).

**Выбор:** **VK ID** (id.vk.com, новый OAuth 2.1 + PKCE + OIDC), а не классический oauth.vk.com. Причины:
- VK постепенно выводит классический OAuth из эксплуатации, новые приложения регистрируются только в VK ID;
- архитектурная консистентность: тот же id_token JWT + JWKS + PKCE pattern, что и для Telegram → переиспользуем `fetchOauthJwks(cacheKey, url)` хелпер;
- встроенная верификация email — VK подтверждает email на своей стороне, у нас сразу `email_verified_at = now()`.

**Что сделано (backend):**
- `AuthController::oauthVkExchange` — POST `/api/auth/oauth/vk/exchange` с code+code_verifier+device_id+redirect_uri.
- Рефактор: `fetchTelegramJwks` → общий `fetchOauthJwks(string $cacheKey, string $url, bool $forceRefresh)` + два wrapper'а. Кэш JWKS в Redis на час с force-refresh при `JWT::UnexpectedValueException`.
- Token endpoint: `POST https://id.vk.com/oauth2/auth` (а не `/token` как у Telegram — это VK-специфика).
- JWKS endpoint: `https://id.vk.com/oauth2/public_keys`.
- iss claim: `https://id.vk.com`, aud = client_id.
- Логика объединения аккаунтов: если у VK-юзера email совпадает с существующим юзером в БД — привязываем VK-identity к нему, не плодим дубли.
- Те же fast-fail настройки сети (5s connect, 15s total, retry 2×500ms) — Yandex Cloud → id.vk.com потенциально может флакать так же как Telegram CDN.

**Что сделано (frontend):**
- `lib/vk-oauth.ts` — start/popSession с PKCE, зеркало `telegram-oauth.ts`. AUTHORIZE_URL=`id.vk.com/authorize`, scope=`email vkid.personal_info`.
- `app/oauth/vk/callback/page.tsx` — обработчик возврата с `?code=...&state=...&device_id=...`. **device_id обязателен** в VK PKCE-flow (отличие от Telegram).
- `lib/auth.ts::oauthVkExchange` — POST к нашему API.
- `app/login/page.tsx`: кнопка VK теперь вызывает `startVkOAuth` (раньше был `alert('OAuth VK — следующий этап.')`).

**Конфиг:**
- [[backend/.env.example|.env.example]]: `VK_CLIENT_ID`, `VK_CLIENT_SECRET`.
- `backend/custom/config-services-vk.snippet` — секция `services.vk` для ручной вставки в `src/config/services.php` на проде. `services.php` не git-tracked — деплоится не через rsync, исторический технический долг (см. снизу).
- Frontend нужен `NEXT_PUBLIC_VK_CLIENT_ID` в `.env.production` (запекается в build).

**Итог (2026-05-16):** VK ID OAuth **работает на проде**. Креды зарегистрированы, применены, юзер успешно вошёл. Но финальная реализация отклонилась от первоначальной — см. следующую запись про user_info.

**Грабли архитектурные (выявились по ходу):**
- `src/config/services.php` правится напрямую на проде, в git только snippet'ы. При смене сервера или при выкатке нового окружения — нужно помнить о ручной правке. **Стоит вынести в `custom/config/services.php`** и синкать через GHA. Записал как технический долг.
- `NEXT_PUBLIC_*` env-переменные **запекаются в build-time** Next.js. Изменение `.env.production` без rebuild не подействует. GHA `Rebuild frontend` справится сам, но при ручной правке `.env` на проде надо запустить `docker compose build frontend && up -d`.

**Грабли пути (общие за день):**
- Process substitution `tee` через `> >(tee -a file)` не успевает дописать буфер при быстром завершении bash → debug-лог в `/tmp/fk_deploy.log` обрезается на половине, хотя workflow реально проходит до конца. Источник истины — `docker events --filter event=restart`.
- `php -r` через ssh — кавычки нужно экранировать тройным слоем (`\"`, `\\\"`, ...). Проще писать через heredoc или scp + exec.

**Известное, отложенное:**
- SMTP 550 на несуществующий ящик (типа `kkk@ru.ru`) проглатывается `safeSendMail` — юзер видит «отправили», но письмо не дошло. Возможный UX: проверять recipient (через `smtp_check` или MX-lookup) до создания verification, либо показывать «не доставлено» через async-проверку статуса.
