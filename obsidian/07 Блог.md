# 07 Блог — детальный план

Модуль блога FK.market: SEO-статьи для органического трафика + роль «журналист»
+ публикация постов в Telegram-канал. Статус: **план на ревью, код не начат.**

Источники: ТЗ заказчика (Google Doc), дополнения владельца — роль «журналист»
(управляет только блогом) и публикация в Telegram-канал.

---

## 1. Доступы и ключи — нужно от владельца

Код не начинаем, пока не готово:

1. **Telegram-бот.** Создать через @BotFather. Токен положить в `.env` бэкенда
   на сервере (`/opt/fk.market/backend/...`) в переменную `TELEGRAM_BOT_TOKEN` —
   **не присылать в чат, не коммитить в репозиторий.** Claude токены не вводит.
2. **Telegram-канал.** Создать/выбрать канал, добавить бота администратором с
   правом «Публикация сообщений». Дать `@username` канала → переменная
   `TELEGRAM_BLOG_CHANNEL`.
3. **Аккаунт журналиста.** Claude не создаёт учётки. После деплоя владелец сам
   заводит пользователя и проставляет ему `role='journalist'` (через `/admin/users`,
   если там есть выбор роли, либо `php artisan tinker`).
4. ⚠️ **Инфра-блокер Telegram.** По [[Bugs]] прод-сервер ненадёжно достукивается
   до Telegram API (egress-прокси в работе, кандидат `37.220.85.24`). Постинг в
   канал будет падать, пока прокси не поднят. Поэтому **Фаза 5 (Telegram) делается
   после закрытия egress-задачи** — или сразу с маршрутизацией через тот же прокси.

## 2. Модель данных

Таблица `blog_posts` (миграция в `backend/custom/migrations/`):

| Поле | Тип | Назначение |
|------|-----|-----------|
| `id` | bigint PK | |
| `slug` | varchar, unique | URL-slug статьи |
| `title` | varchar | H1 и `<title>` |
| `meta_description` | varchar(255), nullable | meta description |
| `excerpt` | varchar(300), nullable | краткое описание для карточки (~150 симв.) |
| `content` | longtext | контент в Markdown |
| `cover_image` | varchar, nullable | путь к обложке 1200×630 (`/storage/blog/...`) |
| `author` | varchar | имя автора (по умолчанию — имя создателя) |
| `tags` | json | массив строк-тегов |
| `related_products` | json | массив slug товаров для CTA-блока |
| `related_posts` | json | массив slug связанных статей |
| `faq` | json | массив `{question, answer}` |
| `status` | varchar, default `draft` | `draft` / `published` |
| `published_at` | timestamp, nullable | дата публикации |
| `telegram_posted_at` | timestamp, nullable | когда запостили в канал |
| `telegram_message_id` | bigint, nullable | id сообщения в канале |
| `created_at`, `updated_at` | timestamp | |

Индексы: `unique(slug)`, `(status, published_at)`.

**Решения по умолчанию** (можно оспорить на ревью):
- `author` — строка, не FK на `users` (как в ТЗ; проще, переживает удаление юзера).
- `tags`/`related_*`/`faq` — JSON-колонки, не отдельные таблицы. Для небольшого
  блога фильтр по тегу = `whereJsonContains`. Если тегов станет много — вынесем в
  `blog_tags` + pivot отдельной задачей.
- FAQ статьи — JSON на посте, отдельно от существующей таблицы `faq_items`
  (та заточена под товары/общий FAQ, другая структура).

## 3. Роль «журналист»

Текущая система ролей бинарная: `role` ∈ {`admin`, `seller`, null}. Добавляем
значение `journalist`. Гранулярную систему прав не вводим — это оверкилл.

- **Новый middleware** `EnsureBlogManagerMiddleware` (alias `blog-manager`):
  пускает `role ∈ {admin, journalist}`, не заблокирован. Существующий
  `EnsureAdminMiddleware` (`admin`) не трогаем.
- **Логин в админку.** `Admin/AuthController::login` сейчас отвергает не-`admin`.
  Меняем проверку на `in_array($user->role, ['admin','journalist'])`.
- **Группировка роутов:**
  - `/admin/me`, `/admin/logout` → `blog-manager` (нужны обеим ролям);
  - `/admin/blog/*` → `blog-manager`;
  - вся остальная админка (`products`, `orders`, `settings`, …) — по-прежнему `admin`.
- **Фронт.** `/admin/me` уже отдаёт `role`. В `app/admin/layout.tsx` массив `NAV`
  получает поле `roles?` на пункт; для `journalist` показываем только «Блог».
  Журналист, зашедший на не-блоговый `/admin/*`, редиректится на `/admin/blog`.

Журналист **может:** CRUD статей, публикация/снятие с публикации, загрузка обложек,
кнопка «Опубликовать в Telegram». **Не может:** товары, заказы, склад, поставщики,
наценки, промокоды, настройки, пользователи.

## 4. Backend

Зеркалит вертикальный срез FAQ-модуля.

- `models/BlogPost.php` — модель, casts (`tags`/`related_*`/`faq` → array,
  `published_at`/`telegram_posted_at` → datetime), scope `published()`,
  `getRouteKeyName() = 'slug'`.
- `controllers/Admin/BlogController.php` — CRUD: `index`, `store`, `show`,
  `update`, `destroy`, `uploadCover` (паттерн `CategoryController::uploadImage`),
  `publishToTelegram`.
- `controllers/BlogController.php` — публичный: `index` (список опубликованных,
  пагинация, фильтр `?tag=`), `show` (статья по slug).
- `routes/api.php`:
  - публичные: `GET /api/blog`, `GET /api/blog/{slug}`;
  - админские (`blog-manager`): `GET/POST /api/admin/blog`, `GET/PUT/DELETE
    /api/admin/blog/{id}`, `POST /api/admin/blog/{id}/cover`,
    `POST /api/admin/blog/{id}/telegram`.
- `services/TelegramChannelService.php` — Фаза 5: постинг в канал через Bot API.

## 5. Frontend — публичная часть

- **`/blog`** — список статей. ISR (`revalidate`), пагинация по 10. Фильтр по тегу
  `?tag=` — клиентский (паттерн `CatalogResults`, который только что внедрили:
  страница статическая, тег применяется на клиенте). Карточка: обложка, заголовок,
  дата, теги, excerpt.
- **`/blog/[slug]`** — страница статьи. ISR (`generateStaticParams` → `[]` +
  `revalidate`, как у товара/категории). Рендер Markdown → HTML с **санитизацией**
  (XSS: контент пишет журналист — полу-доверенный, но чистим обязательно).
  Блоки: контент, CTA связанных товаров (`ProductCard`), связанные статьи,
  FAQ-аккордеон (переиспользуем `FaqAccordion`).
- Новый markdown-рендерер — зависимость фронта (`react-markdown` + `rehype-sanitize`
  или серверный `marked` + `sanitize-html`). Выбор зафиксируем на ревью.

## 6. Frontend — админка

- **`/admin/blog`** — таблица статей (заголовок, статус, дата, теги),
  кнопки создать/удалить. Пункт в `NAV` с иконкой.
- **`/admin/blog/new`** и **`/admin/blog/[id]`** — редактор:
  - Markdown-textarea + живое превью (Фаза 3);
  - загрузка обложки + кроп под 1200×630 (Фаза 3);
  - динамические поля: теги, связанные товары, связанные статьи, FAQ;
  - переключатель `draft`/`published`;
  - автосейв (Фаза 3);
  - кнопка «Опубликовать в Telegram» + индикатор «уже опубликовано».

## 7. SEO

- JSON-LD: `Article` (+`headline`, `image`, `datePublished`, `author`),
  `BreadcrumbList`, `FAQPage` (если есть FAQ).
- `sitemap.ts` — добавить опубликованные статьи (`lastmod` = `updated_at`).
- Мета: `title`, `description`, canonical `/blog/{slug}`, OpenGraph/Twitter
  с обложкой 1200×630.
- Страницы тегов/пагинации — `canonical` на `/blog`, чтобы не плодить дубли.

---

## 8. Telegram-интеграция (Фаза 5)

- Триггер — **кнопка «Опубликовать в Telegram»** в редакторе (выбор владельца:
  ручной контроль, можно повторить).
- `TelegramChannelService::postPost(BlogPost)` — формирует сообщение: обложка
  (`sendPhoto`) + подпись (заголовок + excerpt + ссылка на статью). Формат
  подписи (анонс+ссылка vs полный текст) уточним при реализации.
- Выполнять **через очередь** (Job, как `FulfillOrderJob`) — сетевой сбой не
  блокирует запрос; ретраи. По завершении — `telegram_posted_at` +
  `telegram_message_id`.
- HTTP-клиент маршрутизируется через egress-прокси (см. п.1.4).
- Конфиг: `TELEGRAM_BOT_TOKEN`, `TELEGRAM_BLOG_CHANNEL` в `.env`.

## 10. Открытые вопросы на ревью

1. `author` — строкой или FK на `users`? (план: строка)
2. Теги — JSON-массив или таблица `blog_tags`? (план: JSON)
3. Markdown-рендерер — `react-markdown` или серверный `marked`? (решим в Ф1)
4. Фильтр `/blog?tag=` — клиентский (как каталог) или серверный? (план: клиентский)
5. Формат поста в Telegram — анонс+ссылка или полный текст? (решим в Ф5)
6. Нужна ли журналисту урезанная статистика на дашборде, или сразу `/admin/blog`?
