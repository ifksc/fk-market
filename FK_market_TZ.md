**Онлайн-магазин цифровых товаров на базе API FKwallet**

Версия: 0.6 (MVP)
Дата: 23.04.2026

## 1. Цели и концепция

**FK.market** — онлайн-магазин цифровых товаров с автоматической выдачей после оплаты через платёжный сервис **FKwallet** (Free-Kassa).

**Бизнес-цель v1 (MVP):** запустить работающий магазин, где платформа сама продаёт товары из собственного каталога.

**Бизнес-цель v2+:** превратить магазин в **мультивендорный маркетплейс** — пользователи регистрируются как продавцы, добавляют свои товары, платформа удерживает комиссию.

**Референсы:**
- [plati.market](https://plati.market/) — классический маркетплейс цифровых ключей
- [duck.market](https://duck.market/) — современный UX, фильтры, рейтинги
- [hornybox.ru](https://hornybox.ru/) — ниша цифровых товаров, простая автовыдача

**Ключевая идея:** архитектура сразу закладывает `seller_id` во всех сущностях, чтобы переход v1 → v2 не требовал переписывания БД.

## 2. Типы товаров и способы получения

### 2.1 Типы товаров

| Тип | Типичный способ выдачи | Пример |
|-----|------------------------|--------|
| **Цифровые ключи/коды** | Автовыдача из склада ключей | Steam-ключи, ключи VPN, лицензии ПО, промокоды |
| **Аккаунты** | Автовыдача логин:пароль из склада | Аккаунты стриминговых сервисов, VPN-аккаунты |
| **ИИ-аккаунты** | Автовыдача из склада или через API инвайта | ChatGPT Plus/Team, Claude Pro, Midjourney, Gemini Advanced, Perplexity Pro, API-ключи OpenAI/Anthropic с лимитом |
| **Виртуальные предметы** | Отправка по API / ручная через Steam Trade | Скины Steam (CS2/Dota2), внутриигровые предметы |
| **Подписки и пополнения** | Запрос по API поставщика в момент заказа | Пополнение Steam-кошелька, Spotify, продление VPN |
| **Услуги** | Запрос по API исполнителя / ручная обработка | Накрутка подписчиков, WoW-буст, прокачка в играх |

### 2.2 Способы получения товара (`fulfillment_mode`)

Каждый товар настраивается на один из трёх режимов — режим может комбинироваться с fallback'ом (например, «api → если ошибка, упасть в ручную очередь»).

| Режим | Описание | Когда использовать |
|-------|----------|--------------------|
| **`stock`** (автовыдача из склада) | Ключ/аккаунт берётся из заранее загруженной таблицы `stock_items` и сразу отдаётся покупателю | Ключи игр, ваучеры VPN, заранее купленные аккаунты |
| **`api`** (автовыдача через API поставщика) | В момент оплаты бэк делает запрос к API поставщика (Digiseller, Kinguin, G2G, собственный поставщик) и получает ключ/результат | Пополнения Steam, подписки, накрутки — всё, где есть программный интерфейс |
| **`manual`** (ручная выдача администратором) | Заказ падает в **очередь ручной выдачи**, админ видит задачу, обрабатывает (вводит ключ / выполняет услугу / отправляет трейд) и нажимает «Выдать» | Скины Steam через бот-аккаунт, индивидуальные услуги, fallback при ошибках API |

Правила:
- Для каждого товара админ выбирает основной режим и (опционально) fallback.
- При `api`-режиме: если поставщик ответил ошибкой или таймаутом — заказ автоматически уходит в ручную очередь (не деньги обратно, а задача админу).
- Покупатель на всех режимах видит **одинаковый статус** «обрабатывается / готово» — разница только на стороне админки.
- Каждый товар может требовать **входные параметры** от покупателя (email Spotify, Steam Trade URL, никнейм в игре) — собираются на чекауте и передаются в задачу/API.

## 3. Роли пользователей

1. **Гость** — просмотр каталога, поиск, оформление заказа (без регистрации тоже можно — email для выдачи).
2. **Покупатель** — личный кабинет, история заказов, баланс, повторная выдача купленных кодов.
3. **Администратор** — полный контроль через админку.
4. **Продавец (v2)** — регистрация, добавление товаров, статистика продаж, вывод средств.
5. **Модератор (v2)** — проверка товаров/продавцов перед публикацией.

## 4. Функциональные требования

### 4.1 Витрина (публичная часть)

- **Главная:** баннер, популярные категории, топ товаров, новинки, акции.
- **Каталог:** древовидные категории, фильтры (цена, тип товара, рейтинг, способ выдачи), сортировки (цена, популярность, рейтинг, новизна).
- **Карточка товара:** описание, скриншоты, цена, остаток на складе, рейтинг и отзывы, кнопка «Купить», связанные товары.
- **Поиск:** по названию, описанию, тегам (полнотекстовый, Laravel Scout + MySQL FULLTEXT или Meilisearch).
- **Страницы:** «О нас», «Правила», «Гарантии», «Контакты», «Оферта».

### 4.2 Корзина и оформление заказа

- Добавление в корзину без регистрации (хранится в сессии + cookie).
- Несколько товаров в одном заказе.
- Ввод email (обязательно — туда уходит выдача).
- Для товаров с параметрами (подписки, услуги) — поля ввода на странице оформления (например, логин Steam).
- Применение промокода.
- Выбор способа оплаты (через FKwallet: карты, СБП, крипто — всё что отдаёт FK).
- После оплаты — страница с кодами/аккаунтами + дубликат на email.

### 4.3 Личный кабинет покупателя

- Регистрация по email/телефону + пароль или OAuth: **VK ID**, **Яндекс ID**, **Telegram Login Widget**.
- Профиль: данные, смена пароля, 2FA.
- **История заказов** со статусами: ожидает оплаты / оплачен / выдан / возврат.
- **Повторная выдача:** получить купленный код ещё раз (ключ хранится всегда).
- **Внутренний баланс** (пополняется через FKwallet, может тратиться на покупки).
- **Избранное**, **Отзывы** (только после покупки).
- Поддержка: тикеты/чат с администрацией.

### 4.4 Админка

- **Товары:** CRUD, массовая загрузка ключей (CSV / ручной ввод), управление остатками, категориями, изображениями. Для каждого товара выбирается `fulfillment_mode` (**stock / api / manual**) и при необходимости fallback.
- **Интеграции / Поставщики:** список подключённых внешних API (Digiseller, Kinguin, G2G, собственные). На каждом поставщике: ключи, статус (ok/ошибка), синхронизация каталога (периодическая или по кнопке), маппинг товаров поставщика на товары FK.market, лог запросов и ошибок.
- **Очередь ручной выдачи:** все заказы с `fulfillment_mode = manual` и все заказы, упавшие из `api` (ошибка/таймаут), попадают сюда. Админ видит: что нужно выдать, поля от покупателя (Trade URL, email и т. п.), кнопки «Взять в работу / Выдать / Отменить / Вернуть на платформу». SLA-таймер и алерт, если задача висит больше X часов.
- **Наценки:** глобальная наценка %, наценка по категории, по товару, по продавцу (v2). Автопересчёт отображаемой цены.
- **Заказы:** список с фильтрами, ручная выдача/возврат, просмотр истории платежей.
- **Пользователи:** поиск, блокировка, изменение баланса, назначение ролей.
- **Промокоды:** создание, лимиты, сроки, привязка к товарам/категориям.
- **Статистика (дашборд):**
  - Выручка (день/неделя/месяц/произвольный период)
  - Количество заказов, средний чек, конверсия
  - Топ товаров и категорий
  - Остатки на складе + алерт «заканчивается»
  - Отчёт по наценкам (маржа)
  - График платёжных методов
- **Контент:** редактор статических страниц, баннеров, новостей.
- **Логи:** аудит действий администраторов, вебхуки FKwallet.

### 4.5 Мультивендорность (v2)

- Регистрация продавца, верификация (паспорт/самозанятость).
- Кабинет продавца: добавление товаров, загрузка ключей, просмотр продаж.
- Модерация товаров перед публикацией.
- Автоматический расчёт комиссии платформы (% + фикс).
- Баланс продавца и **вывод средств через FKwallet Payout API** на карту/кошелёк.
- Рейтинг и отзывы на продавца.
- Эскроу: деньги зачисляются продавцу после подтверждения (24-48 ч) или автоматически.

## 5. Интеграции

### 5.1 FKwallet API

- **Приём платежей:** формирование платёжной ссылки, обработка вебхука об успехе/отмене, верификация подписи (HMAC/MD5 по документации FK).
- **Баланс магазина:** запрос текущего баланса платформы.
- **Выплаты (v2):** отправка выплат продавцам на их кошельки/карты.
- **Ключ/секрет** хранятся в `.env`, никогда не попадают во фронт.
- Отдельная учётная запись FK для тестов (sandbox).

### 5.2 Прочие интеграции

- **Email** (SMTP или сервис типа UniSender / Mailgun) — выдача кодов, уведомления.
- **OAuth-провайдеры:** VK ID, Яндекс ID, Telegram Login Widget (через `oauth.telegram.org`). Для каждого — отдельный Laravel Socialite driver.
- **SMS** (например, SMS.ru) — опционально для 2FA.
- **Аналитика:** Яндекс.Метрика (основной), Google Analytics — опционально, цели по ecommerce.
- **Steam API / Trade URL** — для товаров типа «скины»: проверка инвентаря, отправка трейд-оффера покупателю через бот-аккаунт Steam.
- **API поставщиков товаров и услуг** — единый адаптер-интерфейс `ProviderGateway` в Laravel. Методы: `listProducts()`, `getStock($externalId)`, `order($externalId, $params)`, `getStatus($providerOrderId)`. Реализации на каждого поставщика (Digiseller, Kinguin, G2G, собственный API) — можно добавлять без правки бизнес-логики.
- *(Telegram-бот для уведомлений — вне MVP, добавим при необходимости в v2.)*

## 6. Архитектура (high-level)

```
[Frontend: Next.js / React]
           ↓ REST/JSON
[Backend: Laravel API]
     ↓          ↓          ↓
  [MySQL]   [Redis]   [Queue Worker]
                         ↓
                [FKwallet / Email / API поставщиков]
```

- **Фронт:** Next.js на Node.js (SSR для SEO каталога), Tailwind CSS, общий UI-кит.
- **Бэк:** Laravel 11, REST API с JWT/Sanctum, очереди через Redis + Laravel Horizon (выдача, email, вебхуки).
- **База:** MySQL 8 (InnoDB, FULLTEXT-индексы для поиска).
- **Кэш / сессии / очереди:** Redis.
- **Файлы (скриншоты):** S3-совместимое хранилище или VPS + CDN.
- **Админка:** отдельный раздел на том же Next.js или Laravel Nova / Filament (решить на старте разработки).
- **Деплой:** Docker Compose (MVP) → Kubernetes при росте.
- **Продакшн-сервер:** `194.87.118.214` (SSH порт `22`). Все окружения (dev / staging / prod) используют один и тот же Docker Compose с разными `.env`. На сервере: nginx с Let's Encrypt TLS, docker, docker compose, systemd-юниты для рестарта.

## 7. Модель данных (основные сущности)

| Таблица | Ключевые поля | Комментарий |
|---------|---------------|-------------|
| `users` | id, email, phone, password_hash, role, balance, is_blocked | Покупатели и продавцы |
| `sellers` | id, user_id, legal_info, commission_pct, verified_at | Создаётся в v2, в v1 — один «системный» продавец |
| `categories` | id, parent_id, slug, name, sort | Дерево |
| `products` | id, seller_id, category_id, slug, name, description, price_base, markup_pct, fulfillment_type, status, rating | `seller_id` уже есть с v1 |
| `product_images` | id, product_id, url, sort | |
| `stock_items` | id, product_id, payload (зашифрован), is_sold, sold_order_id | Склад ключей/аккаунтов (для `fulfillment_mode=stock`) |
| `providers` | id, code, name, base_url, credentials (зашифр.), status, last_sync_at | Внешние API-поставщики (Digiseller, Kinguin, G2G, …) |
| `provider_products` | id, provider_id, external_id, product_id (nullable), raw_meta (JSON), in_stock, price_in | Маппинг «товар поставщика → товар FK.market»; `product_id` пустой — товар ещё не подключён |
| `fulfillment_tasks` | id, order_item_id, mode (`api`/`manual`), provider_id (nullable), status (`queued`/`in_progress`/`done`/`failed`), assignee_admin_id, input_params (JSON), result (JSON), retries, deadline_at | Очередь выдач: каждая строка — единица работы по позиции заказа |
| `provider_logs` | id, provider_id, request, response, status_code, latency_ms, created_at | Технические логи обращений к API поставщиков |
| `orders` | id, user_id (nullable), email, total, status, payment_id | |
| `order_items` | id, order_id, product_id, qty, price, params (JSON), stock_item_id | |
| `payments` | id, order_id, provider (`fkwallet`), provider_payment_id, amount, status, raw_response | |
| `payouts` | id, seller_id, amount, provider, status | v2 |
| `reviews` | id, product_id, user_id, rating, text, is_approved | Только после покупки |
| `promocodes` | id, code, type (%, fixed), value, limit, used, valid_until | |
| `settings` | key, value | Глобальные настройки (наценка, комиссия и т. п.) |
| `audit_logs` | id, user_id, action, payload, ip, created_at | Аудит админки |

Все денежные поля — `DECIMAL(12,2)`.
Поле `payload` в `stock_items` шифруется на уровне приложения (Laravel `Crypt`).

## 8. Безопасность

- HTTPS обязателен, HSTS.
- Пароли: bcrypt/argon2 (Laravel default).
- 2FA в админке (TOTP — Google Authenticator).
- Rate limiting на API и формах логина/регистрации.
- Защита от CSRF (Laravel), XSS (escape по умолчанию), SQL-инъекций (Eloquent / prepared statements).
- Проверка подписи вебхуков FKwallet — обязательная, с логированием попыток подделки.
- Ключи в `stock_items` хранятся зашифрованными, расшифровка только в момент выдачи.
- Резервное копирование БД ежедневно, шифрование бэкапов.
- Логи действий админов (`audit_logs`) — read-only.
- Соответствие 152-ФЗ: согласие на обработку ПДн при регистрации, политика конфиденциальности, правила хранения.

## 10. Нефункциональные требования

- **Производительность:** главная и каталог — TTFB < 500 мс, SSR-кэш 60 с.
- **Масштаб:** до 100k товаров и 1000 заказов/день без оптимизаций.
- **Доступность:** 99.5% на MVP, 99.9% после v2.
- **Мобильная версия:** адаптив с приоритета mobile-first (у plati.market ~60% трафика с мобильных).
- **SEO:** SSR, человекочитаемые slug-URL, sitemap.xml, JSON-LD для товаров.
- **Локализация:** русский в v1; архитектурно заложить i18n для v2 (английский).

## 11. Риски и открытые вопросы

1. **Блокировка платёжного провайдера** — закладываем возможность подключить второго провайдера (ЮKassa/Robokassa) через общий интерфейс `PaymentGateway`.
2. **Фрод и возвраты** — после выдачи цифрового товара возврат невозможен; в оферте зафиксировать.
3. **Форма ведения деятельности** — заказчик работает как физлицо (без ИП/ООО и ККТ-чеков). Если оборот вырастет — оформить самозанятость / ИП и интегрировать кассу (отложенная задача, вне MVP).
4. **Продавцы-физлица (v2)** — в РФ продажа физлицом в интернете через третьих лиц попадает под НДФЛ. В v2 добавим уведомление продавцу о необходимости самостоятельной уплаты, зафиксируем в оферте.
5. **Модерация контента (v2)** — ручная на старте; автоматизация (эвристики/LLM) — позже.
6. **Скины Steam** — требуют Steam-бота с инвентарём и прошедшего Steam Guard 15-дневного периода; запланировать заранее, если запускаем эту категорию сразу.
